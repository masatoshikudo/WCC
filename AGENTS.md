# エージェント向け制約・ルール・指示

このリポジトリでコードやドキュメントを編集する AI（Cursor 等）および協業者は、次を**必ず**守ってください。

## スコープと変更の原則

- **依頼された範囲だけを変更する。** 無関係なリファクタ、別ファイルの「ついで」修正、スタイルの全面書き換えは行わない。
- **ユーザーが依頼していない変更は行わない（厳禁）。** 例: レイアウトや寸法を「元に戻す」「改善する」、別案に置き換える、依頼と無関係なリセット。依頼に含まれない挙動・見た目・構造を勝手に変えない。曖昧なときは変更せず、確認する。
- **既存の命名・インポート・コンポーネント分割の慣習に合わせる。** 新規コードは同じ作者が書いたように読めること。
- **不要なファイルを増やさない。** ユーザーが明示的に求めない限り、説明用の長文 Markdown やサンプルだけのファイルを追加しない。

## ディレクトリと実装の境界

- **アプリケーションのソースは `site/` 配下。** Next.js のページ・コンポーネント・設定はここに置く。
- **設計の一次情報**は `DESIGN.md`（見た目・トークン・コンポーネント）、`STACK.md`（技術選定・連携）、`VOICE.md`（文言トーン）を優先して参照する。
- **`reference/`** は調査・過去フェーズの資料。仕様の最終決定は `DESIGN.md` / `STACK.md` と実装の整合を取ること。

## UI・フロントエンド

- **Tailwind CSS** でスタイリング。既存のクラス順・ブレークポイントの使い方を踏襲する。
- **アクセシビリティ**（意味のある見出し階層、ボタンとリンクの区別、フォームラベル）を損なわない。
- **ユーザー向け文言**は `VOICE.md` と既存画面のトーンに合わせる（勝手なキャッチコピー量産や敬語の不統一を避ける）。

### ページ内セクションのアンカー命名

- **ランディング等の主要ブロックは `<section>` に意味のある `id` を付ける。** ページ内リンク（`/#…`）、ヘッダー／フッターのジャンプ、計測の区切りに使う。
- **`id` はページ内で一意。** 小文字とハイフンの **kebab-case**（例: `service-detail`, `highlights`）に統一する。
- **固定ヘッダーで見出しが隠れる場合は `scroll-mt-*` を併用**し、既存セクションの付け方に合わせる。
- **`id` を変えたら** `Link href="/#…"` や他ページからのハッシュ参照を **grep で洗い替え**、壊れたアンカーを残さない。トップへのハッシュ `href` は **`site/lib/site-links.ts` の `HOME_ANCHOR_HREF`** に集約し、リダイレクト先（`site/next.config.mjs`）と整合を取る。
- **トップ（ホーム）の既定セット（追加・変更時は整合を取る）:** `hero` → `service-detail` → `highlights` → `what-is-wcc` → `pricing` → `faq`。料金はホームの `#pricing` に集約し、旧 `/packages` は `/#pricing` へ恒久的にリダイレクトする。ギャラリー（9:16 実績相当・お客様の声）はホームの `#highlights` に集約し、旧 `/portfolio` は `/#highlights` へ恒久的にリダイレクトする。

### CSS・レイアウトの安定性（再発防止）

表示崩れや「CSS が外れたように見える」状態は、多くが**削除・置換・誤ったユーティリティ**で起きる。次を**必ず**守る。

1. **ユーティリティ名は推測で書かない。** 色・スペーシングは `site/tailwind.config.ts` の `theme.extend` と一致するクラス名を使う（例: `colors` のキーが `bg-dark` なら `bg-bg-dark`。存在しない `bg-foo` を試さない）。
2. **共有のクラス文字列を編集すると全画面に波及する。** `site/lib/layout/mobile-dock.ts` のような**定数に束ねたクラス**は、ドック・予約フロー等で再利用されている。**依頼なしに見た目の「整理」「軽量化」でプロパティを削らない**（`backdrop-blur`・`border-*`・`overflow-*`・`z-*`・`min-h-*` などは、レイアウトバグ回避のために付いていることがある）。
3. **レイアウト用のユーティリティは「余計」と判断して消さない。** `overflow-x-hidden` / `isolate` / `min-h-[…]` / `pb-24`（固定CTA退避）などは、コメントや近傍コードとあわせて意図を読んでから触る。削る場合は**依頼または明示的なバグ修正**に限る。
4. **近い見た目を別のクラスで代用しない。** フッター帯に寄せたい場合は、`SiteFooter` 等**既に正しく表示されているコンポーネントのクラスをそのまま参照**し、似非の `border-hairline` だけ等に置き換えない。
5. **変更後は `site/` で `npm run build` を通す。** レイアウトや Tailwind を触ったら、可能なら**SP と PC の両方**で当該画面を確認する（`md:hidden` / `md:` の意図が逆になっていないか含む）。
6. **globals.css の `@layer` とクラス名**（例: `.hero-frosted-backdrop`）をリネーム・分割する場合は、**grep で参照箇所をすべて更新**する。
7. **Tailwind の `content`（`site/tailwind.config.ts`）に、`site/lib/**` を含める。** `site/lib/layout/mobile-dock.ts` のように **クラス文字列を定数にだけ置く**場合、ここにパスが無いと `inset-x-0`・`z-40`・任意値 `pb-[calc(...)]` などが **CSS に生成されず**、下部ドックや固定UIのスタイルが欠落する（`className={定数}` は静的解析で展開されない）。

### Next.js: `Cannot find module './NNN.js'`（webpack-runtime / `.next/server`）

- **意味:** 開発サーバーやビルドが **`.next` 内のチャンク一覧**（例: `682.js`）を参照しているが、**実ファイルが無い・古い**。コード欠落ではなく **生成物の不整合**がほとんど。
- **典型原因:** `next dev` の中断、HMR の途中状態、ビルド失敗の直後に再実行、`pull` / ブランチ切替後に古い `.next` が残る、**クラウド同期（Dropbox 等）が `.next` を同期して競合・欠損**する。
- **対処（エージェント・開発者共通の手順）:**
  1. 開発サーバーを止める。
  2. `site/` で **`rm -rf .next`**（必要なら **`rm -rf node_modules/.cache`** も）。
  3. **`npm run dev`** または **`npm run build`** をやり直す。
- **推測でソースを書き換えない。** このエラーが出てもアプリの `app/` や `components/` をいじって解決しようとしない。まず **`.next` のクリーン**を試す。
- **リポジトリ運用:** `.next` は `.gitignore` 済み。**コミット対象にしない。** プロジェクトが Dropbox 等の同期フォルダ内にある場合、同期の対象外に `.next` をする・同期競合時は上記クリーンを優先する。

### `.next` クリーンと本番相当ビルド（エージェントの作業ルール）

次のときは、**ソースを推測修正で追いかける前に** `site/` で **`.next` を削除してから `npm run build`** で再現確認する（開発サーバーは止めてから）。

- **ユーザーから「`.next` をクリーンしてビルド」「続けてクリーン＋ビルド」等と明示されたとき**（依頼どおり **`rm -rf .next`** を実行し、必要なら **`rm -rf node_modules/.cache`** も併用してから **`npm run build`**）。
- **本番相当の検証**（マージ前・大きめのフロント変更後・Tailwind / `content` / 共有レイアウト定数を触った後）。既に「変更後は `npm run build`」とあるが、**同期フォルダ配下では不整合が起きやすい**ため、手戻りや謎のチャンクエラーが続く場合は **クリーンを既定の一手**とする。
- **`Cannot find module './NNN.js'` 等の `.next` 不整合**が疑われるとき（詳細は上記小節のとおり。まずクリーン）。

**Dropbox 等で `rm -rf .next` 後もビルドが `ENOENT`・`rename`・途中で消えたファイル**などになる場合: `site/next.config.mjs` の **`NEXT_DIST_DIR`**（例: **`NEXT_DIST_DIR=.next-local`**）で出力先を同期の外・負荷の少ないパスに寄せる。手順は **`site/.env.example`** のコメントを参照。クリーン時は **既定の `.next` と使用中の `distDir`（例: `.next-local`）の両方**を削除してから `npm run build` する。

**禁止:** クリーンで直る可能性があるのに、**根拠なく** `app/` や `components/` を広く書き換えて「ビルドを通す」こと。

## セキュリティ・シークレット

- **API キー・シークレットをリポジトリにコミットしない。** 環境変数は `.env*.local` 等で管理し、`.gitignore` を尊重する。
- **`.env.example`** にあるプレースホルダだけを共有し、実値は書かない。

## Git・リポジトリ

- **サブモジュール化しない。** `site/` は通常のディレクトリとしてこのリポジトリに含める（ネストした `.git` を作らない）。
- コミットメッセージは**何を・なぜ**が分かる日本語または英語で簡潔に書く。

## 応答・コミュニケーション（ユーザーが日本語を使う場合）

- **ユーザー向けの説明は日本語**で、簡潔かつ実務向けに書く。
- 変更内容を述べるときは、可能なら**ファイルパスと要点**を示す。

## 禁止に近いこと

- **依頼に含まれない編集**（依頼と無関係なレイアウト・寸法の変更、ユーザーが言及していない「元に戻す」など）。
- 依頼と無関係な依存パッケージの大量追加。
- 本番 URL・個人情報・未公開戦略の捏造。
- `node_modules`、`.next`、`.vercel` などビルド生成物やローカル専用ディレクトリを意図的にコミットに含めること。

---

不明点がある場合は、**推測で仕様を足し広げず**、既存ドキュメントとコードを確認するか、ユーザーに確認可能な形で質問する。
