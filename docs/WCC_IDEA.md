## 1. 経緯

- 個人事業として **Wedding Content Creator (WCC)** サービスを立ち上げ予定。
- 結婚式当日の縦型ショート動画を当日中〜翌日に納品するサービス。米国で 2023〜2024 年に急成長した職種を日本市場に持ち込む試み。
- 既にランディングページ (`for-your-wedding-day.com`) を Next.js で公開済み。サイトのコピー・構成は完成している。
- **Tinykomainu の開発手法 (ドキュメント駆動 + Claude Code 連携 + モノレポ構造) が機能していることを確認**し、それを WCC にも適用することで開発速度を上げたいと判断。
- ただし WCC は BtoC リアルサービス業であり、Tinykomainu (BtoB SaaS / インディーハッカー向け) とは顧客層・価格帯・業務フローが根本的に異なる。
- 結論: **別ブランド・別モノレポ。ただしドキュメント駆動の流儀のみ流用**。

## 3. コアバリュー (現時点の言語化)

> **「演出ではなく、当日その場の高まりをそのまま縦型のショート動画に仕立てる」**

既存サイトのコピーを軸に整理:

1. **即時性** — ハイライトは当日中〜翌日に届く。感動が冷めないうちに家族・ゲストとシェアできる
2. **軽さ** — プロのフォト・ビデオが作る重厚な作品の対極。スマホで撮ってスマホで見る、軽やかな記録
3. **補完性** — プロのフォト・ビデオチームと役割が分かれる。記録用と SNS 用を無理なく両立できる
4. **近距離視点** — 「演出を組み立てる」役割ではなく、近い距離から拾う表情やゲストの反応

ただしこれは現状の機能訴求に近い。**Wake における涅槃会モチーフに相当する「世界観の核」は未確定**。着手時にここを掘る必要がある (リスク 7.1 参照)。

## 4. 競合と差別化軸

### 競合カテゴリ

第一に、**プロのウェディングフォト・ビデオグラファー**。価格帯 25〜40万円。長編記録・アルバム制作中心。WCC とは役割が異なるが、新郎新婦の予算枠を奪い合う関係にはなる。

第二に、**式場提携の標準撮影プラン**。多くの場合、新郎新婦は式場経由でこれを契約済み。WCC は式場の標準プランに**追加で**頼まれる位置取り。

第三に、**米国の Wedding Content Creator**。$800〜$2,500 (約 12〜38 万円) で同種サービスを提供。日本国内ではまだ専業プレイヤーが少ない。

第四に、**新郎新婦自身による SNS 投稿 / ゲスト撮影**。コストゼロの「競合」。WCC は「自分たちで撮るより明らかに良い」価値を提示する必要がある。

### 差別化軸候補 (着手時に掘る)

- **当日中〜翌日納品のスピード**: 米国 WCC でも 24-48 時間が標準。日本では希少
- **iPhone × 縦型特化**: プロの一眼カメラで撮った素材を後から 9:16 に切り出すのとは情報量が違う
- **演出に踏み込まない設計**: ドキュメンタリー視点の徹底
- **プロのフォト・ビデオと同時進行できる軽量さ**: 進行を妨げない補完ポジション

## 5. ブランド・メタファー上の位置づけ

Tinykomainu のように「守護獣メタファー = 役割別の犬」というブランド世界観は流用しない。WCC は別ブランド。

ただし Tinykomainu (特に Wake) で確立された **「世界観で記憶される情緒的プロダクト」** という方法論は、WCC に直接応用できると考えている。

WCC が体現すべき情緒の方向性 (現時点の仮説):

- 「結婚式の記録」という重い文化財に対して、**軽やかさ・即時性・親密さ**で対比軸を立てる
- 「永遠に残す」のではなく「**今すぐ分け合える**」記録
- 一生に一度の儀式に、友人視点の親密さを持ち込む

これを Wake における涅槃会モチーフのレベルまで深めるかは要判断 (リスク 7.1 参照)。

## 6. MVP スコープ予想 (着手時に再検討)

### コア機能 (MVP)

**事業者側 (WCC 運営) の業務システム**
- 予約管理 (問い合わせ → ZOOM 打ち合わせ → 見積もり → 決済 → 確定)
- 顧客管理 (新郎新婦の情報、式場情報、当日のスケジュール)
- 納品管理 (撮影日 → 編集 → 納品のステータス)
- 公式 LINE 連携 (新郎新婦との直接やり取りの導線)

**顧客側 (新郎新婦) の体験**
- 既存 LP の維持・改善
- 問い合わせフォーム
- 決済 (Stripe Payment Links)
- 納品物の閲覧・ダウンロード (パスワード保護されたページ等)

### Pro 機能候補 (後回し or プラン分割)

- 複数クリエイターの稼働管理 (将来チーム化する場合)
- 海外挙式対応の見積もり自動化
- 式場別の撮影制約 DB (式場ごとに「外部業者持ち込み NG」「ドローン NG」等を蓄積)

### 意図的に作らない機能 (世界観由来、着手時に再検討)

- ドローン撮影 (機材重装化は「軽さ」と矛盾する)
- 長編フル HD 動画納品 (プロビデオの領域、補完性が崩れる)
- 派手な演出・トレンド BGM (「演出ではなく当日の空気」と矛盾する)
- 大量のオプション追加メニュー (パッケージ単純化を維持)

## 7. 着手時に必ず検証すべきリスク

### 7.1 世界観の核の不在 ★最重要

現時点で WCC には **Wake における涅槃会モチーフに相当する「世界観の核」がない**。コアバリュー (セクション 3) は機能訴求の言語化に留まっており、開発判断の基準として弱い。

将来的に「BGM 派手にして欲しい」「ドローンも撮って欲しい」「長尺フル HD で」等の要望が確実に来る。これに **No と言える根拠** が現状不明確。

着手時の最重要タスク: **コア世界観を一語/一句で言語化する**。候補方向性:
- 「軽さ」の哲学 (重厚な記録に対するカウンター)
- 「即時性」の哲学 (記憶が編集される前の生々しさ)
- 「友人視点」の哲学 (プロの権威ある視点ではなく親密な距離)

これが決まらないとドキュメントの背骨が立たない。

### 7.2 ペルソナの絞り込み不足

現状 LP は「結婚式を控えるカップル」全般に向けている。ペルソナによって価格・機能・コピーが変わる:

- すでにプロのビデオを頼んでる層 vs プロを頼まず WCC だけで済ませたい層
- SNS ヘビーユーザー (自分のアカウントで発信したい) vs 家族親族と共有したいだけの層
- 都心の小規模ウェディング層 vs 地方・式場スタンダード層 vs 海外挙式層

着手時にペルソナを 1〜2 個に絞る。

### 7.3 固有ブランド名・屋号の未確定

「Wedding Content Creator」は職種名であり、固有の屋号ではない。Tinykomainu のように検索可能でユニークなブランド名が必要。

Wake の命名選定経緯のように、**候補と却下理由を残しながら 1 つを選ぶ**プロセスを着手時に実施。

### 7.4 式場との関係構築

日本の結婚式は式場経由の予約が大半。新郎新婦に直接アプローチするだけでなく、**プランナー経由で「外部撮影持ち込み OK」と認められる仕組み作り**が事業継続のカギ。

着手時の検証項目:
- 持ち込み NG の式場の割合 (主要式場チェーンへの調査)
- プランナーへの提案テンプレート作成
- 式場提携プランへの組み込み可能性

### 7.5 法的レビュー

- 特定商取引法に基づく表記 (既存サイトの `/legal/tokushoho` を最新法令に合わせる)
- 結婚式場との撮影権利・肖像権の取り決め
- 新郎新婦・ゲストの肖像権 (SNS 公開時の同意取得フロー)
- 海外挙式時の現地法令 (英国・イタリア・米国等の事例が LP にある)

### 7.6 実績の本物性問題

既存 LP の海外事例 (Lucy and Chris、Sophie & Daniel 等) が**ストック素材を疑われるリスク**がある。日本の事例は「会場名非公開」で具体性が低い。

着手時に:
- 実績アカウント (Instagram/TikTok) の運用開始
- 実際の納品実績の許諾取得・公開
- LP に SNS 実績を埋め込み

### 7.7 競合参入障壁の低さ

iPhone 撮影 + 編集なので機材投資が小さく、真似されやすい。差別化要素を早期に作る必要:
- 編集スタイル (スワイプで覚えてもらえるシグネチャー)
- ブランド (固有名・世界観)
- 特定式場との提携
- 蓄積実績の量と質

### 7.8 業務フロー設計の重さ

Tinykomainu (SaaS) と違い、WCC は **撮影・編集・納品という身体労働**を伴う。1 件 6 時間 + 編集時間 + 移動 + 打ち合わせで、**1人で月に何件こなせるか**が事業のボトルネック。

着手時に:
- 1 件あたりの工数見積もり (撮影 6h + 編集 X h + 移動 + 打ち合わせ)
- 月間処理可能件数の試算
- 価格設計 (15万円 × 月 N 件) の妥当性検証
- スケール時のクリエイター採用・教育設計

### 7.9 SNS 実績アカウントの先行立ち上げ

WCC はサービス特性上、**SNS での作例こそが最大の営業ツール**。LP より先に Instagram/TikTok の実績アカウントが稼働している状態が理想。

LP の SNS アイコンが現状空リンク (`instagram.com/` トップに飛ぶ) だが、これは致命的。着手時に:
- 実績アカウントの開設・運用開始
- LP からの導線張り直し

## 8. Tinykomainu から流用するもの・しないもの

### 流用するもの (開発手法・ドキュメント構造)

- **ドキュメント駆動開発のフロー**: `IDEA.md` → `PRODUCT.md` → `ARCHITECTURE.md` → `DATA_MODEL.md` → `ROADMAP.md` の 5 点セット
- **CLAUDE.md による開発フェーズ宣言**: Claude Code との協業で開発フェーズを明示する慣習
- **Wake 流の「世界観を機能仕様より上位に置く」構造**: 情緒的プロダクトとしての構成原則
- **着手チェックリスト形式**: 開発フェーズごとの ToDo を切り出す形式
- **命名経緯と起源メモを残す習慣**: 半年後の自分が判断軸を再構築できる記録
- **Stripe Payment Links 思想**: ただし WCC は単発決済なのでサブスク基盤は使わない

### 流用しないもの (物理的に不可、または不適合)

- **モノレポ (`pnpm + Turborepo`)**: 別リポジトリ。WCC 単独構成
- **`tinykomainu_*` プレフィックス命名規則**: 名前空間管理が不要
- **`packages/notifications` 等の共通パッケージ**: 別リポジトリなので物理的に流用不可。同じパターン (Resend 経由) は再現可能
- **守護獣メタファー**: WCC は別世界観
- **インディーハッカー向けトーン**: 結婚式を控えるカップル向けのトーンに作り直す
- **サブスクリプション前提のデータモデル**: 単発決済中心の設計に変更

### WCC 固有で新規に必要なドキュメント

- **`WCC_OPERATIONS.md`**: 当日撮影の業務フロー、機材リスト、納品物のフォーマット規定。Tinykomainu はソフトウェア完結だったため不要だったが、WCC は身体労働を伴うので業務マニュアルが必要
- **`WCC_LEGAL.md`**: 結婚式場との持ち込み契約、撮影権利、肖像権、特商法。式場ごとの対応可否リスト・交渉テンプレ
- **`WCC_PRICING_LOGIC.md`**: 交通費・宿泊費の計算ロジック、追加オプション、海外対応の見積もりルール

## 12. 起源メモ (アイデアの原点)

このプロダクトの発想の起点:

1. 米国で 2023〜2024 年に Wedding Content Creator が急成長した職種として確立されているのを認知
2. 日本市場ではまだ専業プレイヤーが少なく、先行者メリットを取れる位置取りを発見
3. 既存のプロウェディングフォト・ビデオ市場と競合せず、補完ポジションが取れる構造の発見
4. iPhone × 縦型 × 即時納品という機材・納期の組み合わせが、SNS 時代の結婚式記録ニーズと完全一致
5. **Tinykomainu の開発手法 (ドキュメント駆動 + Claude Code) が機能していることを確認し、それを別ジャンルに展開する実証実験としても位置付けた** ← 着手の決定打

このアイデアは Tinykomainu とは別の収益源となり、かつドキュメント駆動開発の汎用性検証にもなる。両者は別ブランドとして並立させる。
